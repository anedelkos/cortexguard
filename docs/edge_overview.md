

# Edge Overview

## Purpose
The **Edge service** is responsible for receiving, validating, and processing fused sensor events generated by the simulator or other data sources.  Processing involves running anomaly detection models, executing plans and high level control arbiter.
It acts as the ingestion layer for the simulator for all downstream analytics and model pipelines.

---

## Architecture
The Edge runs as a standalone **FastAPI** service inside a Docker container and communicates with the simulator over HTTP.
In local development, both services share the same Docker network (`kitchenwatch_net`).



+----------------------+     HTTP POST /api/v1/ingest     +-----------------------------+
|   Simulator (src)    | ───────────────────────────────▶ |         Edge API            |
|  (sends fused data)  |                                  |  (validates & processes)    |
+----------------------+                                  +-----------------------------+


## 🐳 Running Simulator and Edge in Containers

For local development, you can launch both the Edge API and the Simulator containers using the Taskfile.

### Start both services
```bash
task edge-sim:up
task edge-sim:run-with manifest=data/manifests/sample_dataset_manifest.yaml rate=1.0 repeat=0 endpoint=http://edge:8080/api/v1/ingest
```


## Logging
* Structured JSON logging by default for container interoperability
* Pretty console output when running locally with LOG_JSON=false
* Expected output:

## Validation
The Edge validates each incoming record using the FusedRecord/WindowedFusedRecord Pydantic model, ensuring:
* Required numeric and path fields are present
* Data types are correct
* No unexpected fields are included
* Invalid records are logged with error-level messages.


## Running on host
```bash
task edge:run
# can be combined with
task simulate:stream
# or
task simulate:local  # with args
```
to run streaming and ingestion outside of docker containers.
